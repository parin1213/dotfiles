# ジャンル別の構成の型

すべて本人の実文書からの観測。冒頭・展開・結びの順で書く。

## ガイドライン

- 冒頭: 「## 目的」または「## なぜこれが必要か」。用語定義が要るなら 💡 コールアウトを本文より前に置く
- 展開: ロール別 or イベント別に同一テンプレを反復（「目的：／参加者：／頻度：／ファシリテーション：」）。「やること」は役割語＋コロン＋内容の箇条書き（「**計画：STORYのTASKへの分解**」）。ルールの背景・例外は ※ かコールアウトで併記。操作はスクリーンショットで示す
- 結び: **まとめを置かずに最後の実務項目でスパッと終わる**。参照は文中リンクで済ませる

## 提案書

- 冒頭の順序: ①目的宣言（自分のロール防衛でなく共通ゴールに接続する） → ②防御解除（「特定の個人の評価を目的とするものではありません」） → ③TL;DR（現状／リスク／提案） → ④たたき台宣言
- 口頭で失敗した議論の仕切り直しなら、失敗の内容を ※ で自分から再現して文書の存在理由にする
- 展開: 相手案・現状施策のメリットを先に公平に書く → デメリット（断定を一段弱める「可能性が高い」） → 自説（肯定→時間軸ずらし「短期的には理解しています。さらに中長期では〜」） → 基本提案は一文で言い切り → 数字付きフォールバック（「暫定的（6ヶ月以内）」「〜に限定する」）
- 事実は表で整理し、未確定は「(案)」「TBD」「？」のまま出す。変更は取り消し線＋色で経緯を残す
- 結び: 測定指標（定量／定性）と正式化条件 → 「最後に」で自己利害の切り離し（「正直瑣末な問題だと考えています」）と信念の言い切り → 「かけていないこと（残論点になりそうなこと）」の自己列挙。一番痛い問い（なぜ今か）も自分で挙げ、答えの種まで置く
- 人事・体制系は付録に感情ケア計画を付ける

### 機能提案（可否を問う短い提案）

上の提案書型のうち、PM 等に「この方向でよいか」を確認する機能提案は次を優先する（2026-07-10 の実指摘から）:

- **結論を冒頭一文の箱で**: 「X を Y します。この方向でよいか確認させてください」が言えれば冒頭はそれで足りる。TL;DR の3ブロックは長文の分析文書用で、可否を問う提案では結論ボックスが勝つ（指摘原文: 「言いたいのは『Admin に生成ボタン付けてやってね。これで良い？』でしょ？前提は丁寧に、でも結論は完結に最初に」）
- **画面・操作を伴う提案には画面イメージを必ず付ける**。状態のバリエーション（未生成／生成済み／対象外 のような分岐）を実データ風に並べ、「見た目は仮。提案は置き場所とふるまいだけ」と注記して UI 議論の早期発散を防ぐ
- **結論に寄与しない節は削る**: 用語解説・効果測定などの説明節は bullet か表に畳む。数字は判断に効く 3 行程度の表へ絞り、残りはリンクへ

## 分析・検証レポート

- 冒頭: 公開意図の一文（「〜に使って欲しいので、公開記事としてまとめます」） → 図 → 「# TL;DR(分析まとめ)」（結論を太字1行） → 言えることの箇条書き → 「## 注意事項」（限界の先出し。「値は確率ではなく相関です」）
- 展開: 目的 → 手法・データ → 表 → 対象ごとに「事実 → 考察」のペア。語尾で区別する: 事実は「〜という結果になった」、主観は「〜と筆者は推測する」「〜の現れのように感じた」
- 結び: まとめ太字1行 → 「現状のデータで分析できないこと」 → 読者と対話するコラム（「平凡な結果だとおもったひとへ: あなたはとても鋭いです」型の、批判的読者を承認してから種明かしする節）

## アジェンダ・会議メモ

- 必ず「目的」から。達成判定できる動詞で書く（「〜の終わらせ方を考えます」「〜が見える」）
- 発散の場では「まとめるのは後日わたしの方で推進します／まずは発散と課題感を乱雑に話したい」とオーナーシップと発散許可を先に宣言。「できるなら：」で欲張りの上限も添える
- 座組を名前＋一行定義で明示（「プロジェクト推進（作業者の作業を阻害する要因があれば取り除く役割）: ◯◯」）
- 用語はその場で平易に定義し「※定義がしたいわけじゃないので、意味の方を重要視して実施します」と定義論争を封じる
- 定例メモは「自分から」→「相手から何かあれば」の二部構成。相手の枠は「あれば」で準備義務を課さない
- アクションアイテムは名前・順序・持ち帰り先まで書いて閉じる。無い回は空欄のまま出す（形式を埋めるための文章を書かない）

## 1on1 メモ（問いかけの型）

- 観測事実＋短い問いを一つ（「苦戦してそうですね。私に出来ることありますか？」）。未来形で聞くと「今困っている」と認めさせる圧がない
- 質問の意図を ※ で注記して圧を抜く（「※改善してほしいという文脈じゃなく、〜なのかを教えてほしいです」）
- 負荷への配慮は三段: 見え方の共有（「〜に見えます」と主観明示）→ 荷下ろし提案 → 手伝う理由
- 先に自分の弱みを開示してから聞く（「進捗どうですか／私はダメです」「お互い吐露しましょう」）
- 聞く目的を明示して材料を集める（「制度化できることはないか検討したいので材料が欲しい」）
- 打診には根拠となる観測と自分側の事情を添える。依頼は協働形（「〜したいので作業協力しませんか」）
- 重い相談には、仮説／事実／結論めいた考え方、と確度ラベルを分けて書き戻し、「一緒に考えれるといいかも？」で共同作業に誘導する
- 不安が予想される局面では安心の先渡し（「色々言われるかもしれませんが、いつも通り基本の型を進めれば良いはずです」）

## PR 本文

- 手書き期の3パターンを案件の性質で使い分ける:
  1. UI機能: 最小記述（1〜2文）＋機能単位のスクショ対列挙。副次情報は details に折る
  2. 調査・修正: 発端リンク → 経緯の番号リスト → 「上記経緯から〜変更しました」で締める
  3. 設計・基盤: 「以下のような使い方を想定して作成しています」と想定利用コードを先に見せる。テーブル定義は Markdown 表
- 現行公認型: やったこと（1段落要約）→ なぜ必要か → 設計判断（攻め筋ごとに1 bullet）→ 変更内容（意味単位3〜6点）→ やらないこと（再検討条件付き）→ 動作確認（未実施は未チェックのまま）→ レビュー ※ で見所をリスクの構造ごと名指し → Note（定量影響）
- レビュー範囲を限定するときは除外先の議論場所と理由をセットで（「スレッド側で議論お願いします。（議論を分散させないためです)」）
- Before/After は2文の対比で書く（「いままでの配布は〜で、配るのも更新も手動でした。プラグイン化すると〜、更新も自動で降ってきます」）

## README・運用ルール

- 構成: 目的 → 役割 → 方針 → 使い分け → 実例 → 参考資料。方針は太字1行＋実在の名前で裏取り
- 規範文の直後に、破ったとき何が起きるかを括弧の口語で即答（「（skip だとハッシュ不変で二度と走らない）」）
- ALWAYS / NEVER 接頭辞＋具体例。禁止には必ず代替を併記
- トラブルシュートは「症状 → まずここを疑う」形式
- 並列関係は散文でなく表で正本化。手順書は冒頭で総量を約束（「**2 ステップ**で復元できる」）
- 「既知の制約」節に放置理由付きで制約を残す。ハマり履歴には「新知見が出たらこの節を更新」と更新義務を書く
- 経緯ナレーション（「旧:」「以前は」）を書かず、現在形の理由だけを書く

## 告知

- 見出しに結論そのもの（「◯◯を用意しました」）
- 意図の説明（「〜という方が多いことがわかったので」）→ 具体的な使い方・実例（実URL付き）
- 結び: 「個人的にお願いしたいこと」と個人の願いとして態度表明し、組織ルールの圧を下げる。失敗談・注意点は付録で開示

## エッセイ・振り返り

- 冒頭: 迷いの告白 → 答えの先出し（「見えてきた、答えのようなものがあります」）→ 文書のロードマップ宣言
- 展開: 時系列の物語。当時の一次資料を引用ブロックで差し込む。決意は独立行の太字。教訓は対比の一行変換で刻む（「評論家から実践者へ。」）
- 結び: 個人の経験を理論に接続 → 冒頭の問いを回収 → 未来への意志
