# サブエージェント委譲と検収

## 委譲時（プロンプトに必ず入れる）

- 作業 worktree の**絶対パス**＋「**それ以外の場所に書くな**」
- 成果物の受け渡し形式（ファイルパス指定。親はファイル経由で回収し、必要部分だけ読む）
- ドメイン用語の定義（confidence 等の多義語は親の解釈を明文で渡す）
- データセット生成系では「**ドメイン知識で生成せず一次ソースを読め**」を明示（スクリプト解析での代替も禁止と書く）
- 走行中の子への SendMessage は届いた保証がない → 仕様変更を送ったら、完了報告時に最新仕様と突合する

## 受領時チェックリスト（採用前に親が実体確認）

### 否定的結論（最重要）
- 「消せない / 必須 / 不可能 / 存在しない」系の結論は**現行コードと実体で裏取り**してから採用する。
  古い ADR・ドキュメント・キャッシュが根拠になっていることが多い（「tesseract.js は消せない」→ 実コードでは消せた、等）。

### 新規テスト
- テストランナーの include 対象に入っているか（vitest の `tests/**` 限定で src/ co-located が silent skip された実績）
- **本体コードを import しているか**（子がロジックを再実装してそれをテストする「空洞テスト」の実績あり）
- fixture は実データ由来か（架空スキーマの fixture で緑のまま本番に出てラベルが常に空だった実績）

### コード・UI
- 参照している CSS トークン・変数・キーが実在するか grep
- i18n がある場合は言語間 parity
- 指定 worktree の外に書いていないか（`git status` を委譲元と main の両方で見る）

### 調査・分析
- 定量的な結論（「45勝3敗」等）は根拠データの実在を確認（LLM 推定のみで実物未確認だった実績 = ハルシネーション事故）
- 数値の転記は原典の訂正情報（固定コメント・正誤表）まで確認

## 原則

子の成果物の品質は**親の担保責任**。「子がやったので」はユーザーへの言い訳にならない。
検収で 1 つでも落ちたら、差し戻しより親が直す方が速いかを判断して対処する。
