# フィードバック PDCA

ユーザーの FB は「その場の直し」で消費すると再発する。**作業修正と規範化を必ず分離**し、
規範化はこの手順で回す。称賛も対象とする（何が効いたかを固定しないと次回再現できない）。

## 手順

### 1. Capture（原文確保）

- FB の**原文を引用のまま**確保する（要約で保存しない。抽象化は次工程）
- 「いま直すもの」（作業修正）と「次から効かせるもの」（規範化）を分けて宣言する

### 2. Generalize（一般規範化）

- 事象を一般規範に抽象化する。**この案件専用のヒントにしない**（過学習禁止。
  評価タスクで規範を直すときは「規範の言い方」の改善だけを行い、正解を教え込まない）
- 禁則を守る: 版ラベル・経緯ナレーション・接地していない語彙を規範文に入れない
- 悪い例/良い例をセリフ単位で添える（実指摘の文言をそのまま悪い例にできることが多い）

### 3. Route（置き場の決定）

上から順に判定する。**先に重複チェック**: 既存規範と同根なら既存項目に統合し、新設しない。

| FB の性質 | 置き場 |
|---|---|
| 全マシン共通の運用規範 | dotfiles `home/.chezmoitemplates/agents-common.md` |
| この仕事 Mac 固有の運用 | dotfiles-work `home/.claude/CLAUDE.work.md` |
| 文書ジャンル・作業種別の規範 | 該当スキル（doc-writing / pr-description-style 等）。無ければ新設を検討 |
| 好み・経緯・PJ 文脈 | memory（`feedback_*.md` + MEMORY.md 索引） |
| 毎回機械的に発動すべき自動化 | hooks（update-config スキル経由。記憶では自動化できない） |

### 4. Apply（反映と配布）

- **管理元（source）を編集する**。実体直編集は禁止（chezmoi-change スキルの手順に従う）
- 配布形態を確認して実体へ反映する（chezmoi apply / symlink なら自動 / コピー配布なら同期）
- コミットする。dotfiles は `mac/master` 運用で local-pc へ渡す（CLAUDE.work.md 参照）

### 5. Check（掃引可能性の検証）

- **本文で説くだけの規範は実行時に落ちる**。スキルなら「提出前セルフチェック」等の
  チェックリストに掃引項目を足したかを必ず確認する
  （実測: チェック項目の追加だけで Opus の規範遵守が 13/15→15/15 になった。2026-07-10）
- 重要な規範は、弱いモデル（サブエージェント）に同じタスクを実行させて再現するか検証する。
  書いた本人には規範文の曖昧さが見えないため、第三者実行が最短の検証になる
  （mizchi の Empirical Prompt Tuning と同型）。採点は「原文・規範文の内側」だけを根拠にする
  （採点者が規範外の知識を持ち込むと誤採点する）

### 6. Act（再発の計測）

- 同種の FB を再度受けたら「規範があったのに効かなかった」＝**規範の書き方の失敗**として扱う。
  規範の存在を言い訳にせず表現を書き直し、該当規範に事例と日付を追記して再発回数を可視化する

## 報告様式

規範化の報告は 4 行で:

```
FB: <原文の要点>
規範: <一般化した規範を一文で>
反映先: <ファイル / コミット / 配布状態>
検証: <チェックリスト化の有無・第三者実行の結果 or 未検証と明記>
```
