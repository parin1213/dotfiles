# 夜間自走の運用

## 完遂規範（最優先）

- 委譲された作業は**完遂する**。「対象が枯れた」「判断が必要」を理由に自己終了しない。
  実績: /loop の自己終了 ×4 → ユーザーが就寝中に同一プロンプトを 5 回再投入、「なんでやってないねん」「出来ない言い訳は聞きたくない」。
- 委譲された判断は持ち帰らない。設計フォークはデフォルト案で進め、採用案を STATE.md に 1 行記録する。
- 詰まった案件も放置せず「枠組み・調査・PR 下書きまで」進めて次へ移る。
- ユーザーに残すのは①実機検収②仕様判断③prod リリース可否のみ。「レビュー待ち」で成果物を積まない
  （コードの正しさはこちらの担保責任。マージまで進める — プロジェクト規約の merge 手順があればそれに従う）。

## rate limit 監視

- 使用率: `~/.claude/statusline.log` を grep（例: `grep -o "5h [0-9]*%\|7d [0-9]*%" ~/.claude/statusline.log | tail -2`）。
- **statusline は更新遅延がある**。リセット直後は小型エージェント 1 体を probe に出し、生きて返ったら本隊再開。
- 閾値: 5h ≥93% → 新規着手停止・進行中をきれいに締めて wakeup 待機。7d ≥90% → 停止してユーザーに相談（自動再開禁止）。
- リセット時刻は可変。固定アンカーから外挿しない。ユーザー申告（「次のリセット22:40だってさ」）が来たら即更新。
- 時刻をファイルに記録するときは**必ず日付付き**（時刻のみだと翌日に計算不能）。

## STATE.md（作業ディレクトリ直下に常備）

毎サイクル更新する。必須項目:

```markdown
# STATE (2026-07-07 03:12 更新)
## 完了: <PR/コミット単位で>
## 進行中: <何をどこまで・次の一手>
## 判断ログ: <デフォルト採用した設計フォーク 1 行ずつ>
## resume 手順: <新セッションがこのファイルだけで再開できるコマンド・文脈>
## limit 状況: 5h N% / 7d N%（YYYY-MM-DD HH:MM 時点）、想定リセット <日付+時刻>
```

## 再開の多重フォールバック

ScheduleWakeup は **limit 中に発火すると即死する**。単独に頼らない:

1. wakeup をリセット想定時刻＋5分に設定（`prompt` に STATE.md のパスを含める）
2. 死んだ場合に備え、STATE.md の resume 手順は「ユーザーが手動で『つづけて』と打つだけで同じ地点から再開できる」水準に保つ
3. 自己 nudge スクリプト（SendKeys 等）は分類器が自己起動を拒否する → 必要ならユーザーに起動を依頼する

## 朝の引き継ぎ報告

冒頭に **[判断待ち]** と **[作業依頼（実機検収等）]** を分けて箇条書き（release-handoff スキルの様式）。
続けて: 完了一覧（反映先つき: merge 済み / staging / prod）→ 未検証範囲 → 判断ログ。
平易な日本語で書く（/compact 直後は文体が崩れやすい。「日本語になってない」と言われた実績あり）。
