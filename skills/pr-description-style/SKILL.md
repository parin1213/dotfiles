# PR 説明文の文体

日本語 PR 本文の規範。一般的な文章規範は japanese-tech-writing に従い、PR ではさらに以下を優先する。

## 節構成

`なぜ / 仕様 / 変更内容 / 確認内容 / 制約条件` の 5 節。

- 「設計判断」ではなく「仕様」。「テストプラン」ではなく「確認内容」。「既知の制約」ではなく「制約条件」
- 全節ですます調。箇条書きは体言止めとの混在可

## なぜ: 目的を一文で先頭に

「なぜ」節は目的（この変更で何ができるようになるか）を一文で冒頭に置き、経緯や現状の問題はその後に回す。

悪い例（経緯から始める）:

> いままでの配布は ZIP を各自取り込む方式で、更新も手動でした。（…3文…）どこでも使えるようになるのも大きいです。

良い例（目的が先頭）:

> web チャット、Desktop の Chat タブ、Cowork、Claude Code のどこでも同じスキルが使えるようにするためです。
>
> いままでの配布は ZIP を各自取り込む方式で、更新も手動でした。

## 仕様: 自己正当化を書かず、結果を言い切る

設計の弁明（「〜する理由がなかったです」「〜なのでこうしました」の連発）を書かない。変更後にどうなるかを仕様として言い切る。

- 悪い例: 「7 スキルは親子セットで動くものなので、複数プラグインに分ける理由がなかったです」
- 良い例: 「リポジトリごと 1 プラグイン `ca-matching` になります。」

根拠が複数あるときは、親の主張の下に子の箇条書きでネストする（1 行に詰め込まない）:

```
- 配布は組織マーケットプレイス経由の想定。
  - 組織マーケットプレイスは private / internal リポ限定という公式の縛りがあり、このリポは private なので条件を満たします。
  - 手順と公式ドキュメントの根拠は README に書きました
```

## 確認内容: 条件形のチェックリスト

「〜が通ること」「〜であることを確認」の条件形で書く。実行結果の報告調にしない。

- 悪い例: 「ユニットテスト 8 本 green」「両方で pass」
- 良い例: 「ユニットテストが通ること」「`claude plugin validate --strict` が通ること」

件数などの装飾的な数値も、確認観点として意味がある場合だけ残す。

## 語彙

- 英語口語（green / pass / WIP 的なジャーゴン）は使わない
- 比喩がかった口語も使わない（「そのまま乗れます」→「条件を満たします」）
- 平易な口語表現は可（「更新も自動で降ってきます」程度）

## 制約条件: 残タスクは実施タイミングを言い切る

「〜側で別途」のような宙吊りにせず、いつ誰がやるかを書く。

- 悪い例: 「Secrets の削除はリポジトリ設定側で別途」
- 良い例: 「Secrets の削除をマージ次第実施します。」
