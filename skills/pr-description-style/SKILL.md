# PR 説明文の文体

日本語 PR 本文の規範。一般的な文章規範は japanese-tech-writing に従い、PR ではさらに以下を優先する。

読者モデル: **人間は「結論・前提 → やったこと → 意図」の順でしか理解できない**。何の話か・なぜ必要かを先に固定できないと、詳細を読んでも保持できない。Description だけ読めば全体像が頭に入り、そこからソースを見に行ける「案内」になっているかで品質を判定する。

構成を考える前に、**このPRを主に誰に見てもらい、何を判断してもらうのか**を一文で決める。複数の読者がいる場合は、主な読者の判断材料を前半で閉じ、実装担当者だけが必要とする詳細を後半へ置く。全員へ同じ詳しさで説明しない。

## 節構成

`なぜ / 仕様 / 変更内容 / 確認内容 / 制約条件` の 5 節。

- 「設計判断」ではなく「仕様」。「テストプラン」ではなく「確認内容」。「既知の制約」ではなく「制約条件」
- 全節ですます調。箇条書きは体言止めとの混在可

複数の職種が読む長いPRでは、5節の役割を保ったまま、読者が知っている言葉で節を分けてよい。

- 冒頭で、主な読者と確認してほしい判断を明記する。
- 前半は、主な読者が判断に使う「目的、実測した効果、業務上の変更範囲、残る問題」だけで閉じる。
- 後半は、実装担当者が確認する「処理の構成、導入方法、変更箇所、検証結果」を置く。
- 前半の見出しに、ファイル名、内部Phase名、CLI名など、主な読者が判断に使わない実装語を出さない。
- 「PdM向け」「エンジニア向け」のような読者ラベルだけを見出しにしない。「待ち時間と品質への効果」「業務フローの変更範囲」のように、その節で扱う対象を見出しにする。

## なぜ: 目的を一文で先頭に

「なぜ」節は目的（この変更で何ができるようになるか）を一文で冒頭に置き、経緯や現状の問題はその後に回す。

**課題が複数あるなら、冒頭で「N 個の課題を解決します」と宣言し、`課題1:` `課題2:` の見出しで独立させる。** 仕様側の先頭で「課題1 は◯◯で、課題2 は△△で解決します」と対応付ける。実測や背景の列挙（「前提となる実測が 2 つあります」）は課題の宣言にならない（2026-07-10, grooves/digester#18 で指摘）。目的を一文に圧縮した結果、複数課題の骨格が消えるのが典型的な事故。

悪い例（経緯から始める）:

> いままでの配布は ZIP を各自取り込む方式で、更新も手動でした。（…3文…）どこでも使えるようになるのも大きいです。

良い例（目的が先頭）:

> web チャット、Desktop の Chat タブ、Cowork、Claude Code のどこでも同じスキルが使えるようにするためです。
>
> いままでの配布は ZIP を各自取り込む方式で、更新も手動でした。

### 性能・コスト改善は結論を最初の画面に置く

性能・コスト・作業時間を改善する PR は、目的文の直後に**変更前 → 変更後の絶対値、差、率、読者にとっての意味**を置く。詳細を読まないと改善量が分からない構成にしない。入力件数など比較条件が違う場合は同じ表で明記し、率を出せない指標は「比較不能」と書く。

- 悪い例: 「複数のボトルネックを調査し、並列化と計測を追加しました」
- 良い例: 「最長工程を49分26秒から21分08秒へ短縮しました。28分18秒、57.2%の短縮です」

## 仕様: 自己正当化を書かず、結果を言い切る

設計の弁明（「〜する理由がなかったです」「〜なのでこうしました」の連発）を書かない。変更後にどうなるかを仕様として言い切る。

- 悪い例: 「7 スキルは親子セットで動くものなので、複数プラグインに分ける理由がなかったです」
- 良い例: 「リポジトリごと 1 プラグイン `ca-matching` になります。」

根拠が複数あるときは、親の主張の下に子の箇条書きでネストする（1 行に詰め込まない）:

```
- 配布は組織マーケットプレイス経由の想定。
  - 組織マーケットプレイスは private / internal リポ限定という公式の縛りがあり、このリポは private なので条件を満たします。
  - 手順と公式ドキュメントの根拠は README に書きました
```

## 散文を構造に変換する（文字文字対策）

説明が散文 bullet の連続になったら（=文字文字してる）、次の変換を機械的に適用する:

| 説明対象 | 変換先 |
|---|---|
| データの形（レスポンス・設定・スキーマ） | **実例をコードブロックで冒頭に**。文章で形を説明しない |
| 分岐・順序のある処理 | **テキスト図**（`→` `├` で描く）。箇条書きの入れ子にしない |
| 変更ファイルが 4 つ以上 | **表**（ファイル / 内容 / 読みどころ）。「読みどころ」列で読み順を案内する |
| 3 文以上ある bullet | 親の言い切り 1 文 + 子のネスト根拠に分解 |
| 設計の弁明が続く段落 | 結果の言い切りに圧縮し、経緯の詳細はコードコメントやリンク先へ逃がす |

散文 bullet が 5 個以上連続したら、上のどれかに変換できないか必ず点検する。

## 提出前セルフチェック（読み手テスト）

書き終えたら、読者になりきって次を自問する。1 つでも No なら直す:

- Description だけ読んで（コードを見ずに）「何が変わるか」を 1 文で言えるか
- このPRを主に誰に見てもらい、何を判断してもらうのかが冒頭に書かれているか
- 複数の読者がいる場合、主な読者が前半だけで判断でき、実装担当者だけが必要な詳細は後半へ分かれているか
- 各見出しは、その位置まで読んだ人が知っている言葉だけで内容を特定できるか
- 性能・コスト改善なら、最初の画面だけで変更前後の絶対値・差・率・読者への意味が分かるか。比較条件の違いを隠していないか
- 課題と解決策の対応が明示されているか（複数課題なら番号で紐づいているか）
- **観測 → そこから言える課題 → 目標 → 変更 → 同条件の確認 → 残る制約**の順に読めるか。観測から結論へ移る各段に根拠があり、書き手の思い込みで因果関係を補っていないか
- どのファイルから読み始めればよいか書いてあるか
- 「まず〜して、次に〜した」という**作業ログ視点のナレーションが残っていないか**（読者に必要なのは結果の構造。書き手の時系列ではない）
- **書き手が知っていることを全部書いていないか**。各節で読者が得るべきことは 1 つ。寄与しない詳細は削るか、コードコメント・リンク先へ移す（詳しさは親切ではない。読み手の認知コストが品質）
- 確認内容（動作確認）が**すべて条件形**か。実行結果の報告調（「10 件実行し 6 件成功」）が地の文に残っていたら、条件形の親＋根拠のネストに変換したか
- 設計理由が**「〜したのは〜ためです」の弁明形で連続していないか**。「結果の言い切り＋理由のネスト」に変換したか（例: 「保存先を X にしたのは Y のためです」→「保存先は X です。」＋ネストで「Y のためです」）
- **表・図に写した事実の精度が原文と一致しているか**。件数・本数・対象の単複を要約で丸めていないか（migration 2 本を「migration」1 行に畳む等）。構造化は圧縮ではない
- **リリース TODO・制約の各項目は、いま時点でも未実施か確認したか**。済んだものが「これからやります」のまま残っていないか（転記でなく検証）
- **他 PR の状態（Approve 済み等）や所与の前提の再説明を書いていないか**。依存は順序の事実だけになっているか

## 確認内容: 条件形のチェックリスト

「〜が通ること」「〜であることを確認」の条件形で書く。実行結果の報告調にしない。

- 悪い例: 「ユニットテスト 8 本 green」「両方で pass」
- 良い例: 「ユニットテストが通ること」「`claude plugin validate --strict` が通ること」

件数などの装飾的な数値も、確認観点として意味がある場合だけ残す。

## 語彙

- 英語口語（green / pass / WIP 的なジャーゴン）は使わない
- 比喩がかった口語も使わない（「そのまま乗れます」→「条件を満たします」）
- 平易な口語表現は可（「更新も自動で降ってきます」程度）

## 制約条件: 残タスクは実施タイミングを言い切る

「〜側で別途」のような宙吊りにせず、いつ誰がやるかを書く。

- 悪い例: 「Secrets の削除はリポジトリ設定側で別途」
- 良い例: 「Secrets の削除をマージ次第実施します。」

## 環境・外部の状態は「転記」でなく「検証」して書く

PR 本文が言及する外部状態（設定済みの credentials・デプロイ状況・依存 PR）は、**書く時点の実態を確認してから書く**。既存本文や過去の情報の転記は、実態とズレた瞬間に読者を誤導する。

- **リリース TODO・残タスクは、書く直前に「まだ未実施か」を確認する**。既に済んだ項目を「これからやります」と書かない（済んだ項目は消すか、残りのスコープだけに絞る）。実施済みかは観測で判定できることが多い（例: staging で疎通している＝staging の設定は済んでいる）
- 検証できない状態は断定せず「（未確認）」と書くか、確認を依頼する
- **他 PR の状態ナレーション（Approve 済み・レビュー中・マージ待ち等）を書かない**。状態はすぐ古びる。依存は「#N のデプロイを本 PR より先行させます」のような**順序の事実**だけを書く
- 既に所与になった前提の再説明はノイズ（依存先の中身の解説は依存先の PR に任せ、参照だけ置く）
