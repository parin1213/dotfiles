# 実機検収ハンドオフの様式

## 依頼前の自己検証（これを通してから依頼する）

1. 変更が**検収対象の環境に実際に載っているか**を確認する（デプロイ済みバンドルの中身・バージョン・タグ）。
   未デプロイのまま検収依頼した実績あり（「＞少なくともStagingでは太字になっていないよ」）。
2. 自分で通せる主経路は先に通す（実クライアントと同一パラメータの疎通・スクショ）。
   ユーザーを計測器にしない: 同じ不具合の実機再テストを 2 回依頼したら、3 回目の前に必ず計測（ログ・デバッグ出力）を仕込む。

## チェックリスト様式

```markdown
## 検収お願いします（反映先: staging / prod / desktop 0.4.1 など明記）

前提: <検収に必要な状態。例: auto-mode ON / 完全終了（トレイ右クリック→終了）してから再起動 / ログイン済み>

1. <操作> → <期待結果>
2. <操作> → <期待結果>
...

未検証: <こちらで確認できていない範囲>
```

- **操作→期待結果**の番号付きで書く。「見ておいてください」のような自由確認にしない。
- 前提条件（モード・再起動方法・アカウント）を先頭に。前提欠けの空振りが複数回あった。
- **スマホで空き時間に検収する**前提を意識する（項目は少なく・判定は目視 1 秒で済む形に）。
- 結果は「ちゃんと動作したよ」の一言で返ってくる。NG 時はログ・スクショが貼られるので、
  貼られたものだけで診断が始められるよう、期待結果は具体的に書いておく。

## 報告との関係

完了報告の冒頭「[作業依頼]」にこのチェックリストへの参照を置く。判断待ち（仕様・リリース GO）とは分離する。
